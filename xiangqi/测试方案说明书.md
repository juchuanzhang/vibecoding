# 象棋分析软件（XiangQi Analyzer）测试方案说明书

## 1. 引言

### 1.1 编写目的

本说明书定义象棋分析软件的测试策略与测试方案，确保系统在功能正确性、规则完整性、计算准确性、性能达标和用户体验等方面满足需求规格要求。

### 1.2 适用范围

覆盖 Rust 核心引擎（WASM）、前端界面、Web Worker 通信等全部模块。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 单元测试 | 针对单个函数/方法的测试 |
| 集成测试 | 钨对多模块协作的测试 |
| 系统测试 | 钨对完整系统的端到端测试 |
| 回归测试 | 修改后重新执行已有测试，确保未引入新缺陷 |
| 性能测试 | 验证系统响应时间和吞吐量是否达标 |
| Alpha-Beta | Minimax搜索优化算法 |
| FEN | 棋局状态标准文本格式 |
| WASM | WebAssembly，浏览器可执行字节码 |

## 2. 测试策略

### 2.1 测试层级

采用 **V模型** 测试策略，从底层到顶层逐级验证：

```
        ┌────────────┐
        │  系统测试   │  ← 验证整体功能
        ├────────────┤
        │  集成测试   │  ← 验证模块协作
        ├────────────┤
        │  单元测试   │  ← 验证函数逻辑
        ├────────────┤
        │  代码审查   │  ← 验证代码质量
        └────────────┘
```

### 2.2 测试类型分配

| 测试类型 | 占比 | 执行阶段 | 主要目标 |
|---------|------|---------|---------|
| 单元测试 | 50% | 开发阶段 | 验证函数/算法逻辑正确性 |
| 集成测试 | 20% | 开发完成后 | 验证模块间接口与协作 |
| 系统测试 | 15% | 全部开发完成后 | 验证端到端用户流程 |
| 性能测试 | 10% | 系统测试后 | 验证响应时间与搜索深度 |
| 兼容性测试 | 5% | 系统测试后 | 验证跨浏览器运行 |

### 2.3 测试工具

| 测试对象 | 工具 | 说明 |
|---------|------|------|
| Rust 核心引擎 | `cargo test` | Rust 内置测试框架 |
| WASM 导出接口 | `wasm-pack test --headless` | 浏览器环境下的 WASM 测试 |
| 前端 JS | 手动测试 + 浏览器开发者工具 | JS 暂不引入测试框架，以手动为主 |
| 性能基准 | `cargo bench` | Rust 性能基准测试 |
| 浏览器兼容性 | Chrome / Firefox / Edge / Safari 手动验证 | 跨浏览器测试 |

### 2.4 测试原则

1. **规则优先**：走棋规则是核心，必须100%正确，任何规则错误均为严重缺陷
2. **性能底线**：4层搜索必须在2秒内返回结果，否则为高优先级缺陷
3. **评估合理性**：评分结果必须符合象棋常识（如车比马贵），明显不合理为缺陷
4. **UI可用性**：所有交互必须可用，但视觉细节为低优先级
5. **自动化优先**：Rust 引擎测试尽可能自动化，前端以手动测试为主

### 2.5 缺陷等级定义

| 等级 | 定义 | 示例 | 修复时限 |
|------|------|------|---------|
| P0-致命 | 规则引擎错误、系统无法运行 | 蹩马腿未实现、WASM无法加载 | 立即 |
| P1-严重 | 核心功能异常、性能不达标 | 搜索结果明显错误、4层搜索超5秒 | 24小时 |
| P2-一般 | 非核心功能异常、体验不佳 | 悔棋偶尔失败、推荐箭头显示偏差 | 3天 |
| P3-轻微 | 视觉细节、非功能性问题 | 棋子字体大小、颜色微调 | 后续版本 |
