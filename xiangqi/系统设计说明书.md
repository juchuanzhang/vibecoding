# 象棋分析软件（XiangQi Analyzer）系统设计说明书

## 1. 引言

### 1.1 编写目的

本说明书详细描述象棋分析软件的系统架构、模块设计、数据结构、接口定义及算法设计，为开发人员提供技术实现指导，为测试人员提供验证依据。

### 1.2 系统概述

XiangQi Analyzer 是一款基于 Web 的中国象棋分析软件，采用 Rust + WASM 作为高性能引擎核心，HTML/CSS/JS 作为前端界面，实现象棋走棋、局势评分与最优走法推荐功能。系统全部在浏览器端运行，无需后端服务器。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| WASM | WebAssembly，浏览器可执行的低级字节码格式 |
| wasm-pack | Rust 官方 WASM 打包工具，生成 JS 绑定和 WASM 文件 |
| FEN | Forsyth-Edwards Notation，棋局状态的标准文本表示格式 |
| Alpha-Beta 剪枝 | Minimax 搜索的优化算法，通过剪枝减少不必要的节点评估 |
| Transposition Table | 置换表，哈希缓存已评估局面，避免重复计算 |
| Zobrist Hashing | 基于随机数的棋局哈希方法，用于置换表键值 |
| Web Worker | 浏览器后台线程，用于异步执行计算任务 |
| PVP | Player vs Player，人对人模式 |
| PVA | Player vs AI，人对AI模式 |

## 2. 系统架构设计

### 2.1 总体架构

系统采用 **三层架构**：引擎层（Rust/WASM）、中间层（Web Worker）、界面层（HTML/JS）。

```
┌─────────────────────────────────────────┐
│                 界面层                    │
│  HTML / CSS / JavaScript (DOM 操作)      │
│  board.js  game.js  main.js              │
├─────────────────────────────────────────┤
│                 中间层                    │
│         Web Worker (worker.js)            │
│   异步消息传递 ─→ WASM 调用               │
├─────────────────────────────────────────┤
│                 引擎层                    │
│        Rust → WASM (xiangqi-core)        │
│  board.rs  rules.rs  move_gen.rs          │
│  engine.rs  evaluation.rs  types.rs       │
└─────────────────────────────────────────┘
```

### 2.2 通信机制

界面层与引擎层通过 **Web Worker 消息传递** 通信，避免 WASM 计算阻塞 UI 渲染：

```
┌──────────┐   postMessage(请求)   ┌──────────┐   WASM调用   ┌──────────┐
│  main.js │ ────────────────────→ │worker.js │ ──────────→ │ WASM引擎 │
│ (UI线程) │                       │(后台线程) │              │ (Rust)   │
│          │ ←──────────────────── │          │ ←────────── │          │
└──────────┘   postMessage(结果)   └──────────┘   返回结果   └──────────┘
```

**消息格式**：

请求消息（UI → Worker）：
```json
{
  "type": "analyze | validate_move | get_score | new_game | import_fen",
  "data": { ... }
}
```

响应消息（Worker → UI）：
```json
{
  "type": "analysis_result | validation_result | score_result | ...",
  "data": { ... }
}
```

### 2.3 WASM 导出接口设计

Rust 通过 `wasm_bindgen` 导出以下接口供 JS 调用：

    pub fn from_fen(fen: &str) -> Result<XiangQiEngine, JsValue>;

    pub fn to_fen(&self) -> String;

    pub fn evaluate(&self) -> i32;

    // 执行走棋，返回是否合法
    pub fn make_move(&self, from_x: u8, from_y: u8, to_x: u8, to_y: u8) -> bool;

    pub fn undo_move(&mut self) -> bool;

    pub fn get_legal_moves(&self, x: u8, y: u8) -> Vec<Position>;

    pub fn is_in_check(&self) -> bool;

    // 检测当前是否胜负已定
    pub fn is_game_over(&self) -> Option<u8>; // 0=红胜, 1=黑胜

    // 获取走棋历史

### 2.4 数据结构设计

```rust

// 棋子方
#[repr(u8)]
pub enum Side {
    Red,
    Black,
}

// 棋子
pub struct Piece {
    piece_type: PieceType,
    side: Side,
}

// 棋盘（9×10 = 90个位置）
pub struct Board {

    black_king_pos: Position,     // 黑将位置缓存

pub struct Move {
    from: Position,
    to: Position,

pub struct BoardState {

    last_move: Option<Move>,

#[wasm_bindgen]
pub struct PieceInfo {

    y: u8,

pub struct MoveRecord {
    move_number: u32,

    to: Position,
}
```

## 3. 模块详细设计

### 3.1 board.rs — 棋盘数据结构模块

**职责**：棋盘表示、棋子操作、状态查询

**关键设计**：

| 设计项 | 方案 | 原因 |
|--------|------|------|
| 棋盘表示 | 一维数组 `[Option<Piece>; 90]` | 内存紧凑，索引快速（y*9+x） |
| 帅/将位置缓存 | `red_king_pos` / `black_king_pos` | 避免每次遍历查找，飞将检测O(1) |
| 哈希方案 | Zobrist Hashing | 增量更新O(1)，支持置换表 |
| 增量走棋 | `make_move` 修改棋盘 + 记录被吃棋子 | 快速走棋/撤销 |
| 增量撤销 | `undo_move` 恢复棋盘 + 恢复被吃棋子 | 高效悔棋 |

**Zobrist 哈希设计**：
- 预生成随机数表：10行 × 9列 × 7棋子类型 × 2方 = 1260个64位随机数
- 初始哈希 = 所有棋子位置随机数之和 + 走棋方随机数
- 走棋时增量更新：hash ^= zobrist[from] ^= zobrist[to] ^= zobrist[captured]

**棋盘坐标约定**：
```
(0,0) ──── (8,0)   黑方底线（行0）
  │           │
(0,4) ──── (8,4)   河界
  │           │
(0,9) ──── (8,9)   红方底线（行9）
```
- x: 0-8（列），y: 0-9（行）
- y=0-4 为黑方区域，y=5-9 为红方区域

### 3.2 rules.rs — 走棋规则模块

**职责**：各棋子走法规则实现、将军检测、合法性验证

**棋子走法实现**：

#### 帅/将
- 目标必须在九宫内：红方 (3-5, 7-9)，黑方 (3-5, 0-2)
- 每次移动一步（上下左右）
- **飞将规则**：帅将不能在同一列上无棋子阻隔对面

```
飞将检测算法：
1. 若帅将同列（x相同）
2. 遍历两帅将之间所有位置
3. 若存在任一棋子 → 合法（有阻隔）
4. 若无棋子 → 非法（飞将）
```

#### 仕/士
- 目标必须在九宫内
- 斜走一步（dx=1, dy=1）

#### 相/象
- 走田字（dx=2, dy=2）
- 不能过河：红方相 y≥5，黑方象 y≤4
- **塞象眼检测**：田字中心位置（from.x+dx/2, from.y+dy/2）有棋子 → 非法

#### 马
- 赌马步：先直走一步再斜走一步
- 8个可能方向：(±1,±2) 和 (±2,±1)
- **蹩马腿检测**：直走那一步的位置有棋子 → 非法
  - 如目标 (x+2, y+1)，蹩腿位为 (x+1, y)
  - 如目标 (x+1, y+2)，蹩腿位为 (x, y+1)

#### 车
- 直线移动（横/纵）
- 路径上不能有棋子阻隔
- 可吃目标位置的敌方棋子

#### 炮
- 不吃子时：同车，直线移动，路径无棋子
- 吃子时：路径上必须恰好隔一个棋子（炮架），目标为敌方棋子

#### 兵/卒
- 红兵：未过河(y≥5)只能前进(y-1)；过河(y≤4)可前进(y-1)和左右(x±1)
- 黑卒：未过河(y≤4)只能前进(y+1)；过河(y≥5)可前进(y+1)和左右(x±1)
- 不能后退

#### 合法性验证流程
```
1. 检查起点是否有己方棋子
2. 检查目标位置是否在棋盘范围内
3. 根据棋子类型检查走法是否符合规则
4. 检查目标位置是否为己方棋子（不能吃己方）
5. 临时执行走棋
6. 检测走棋后己方是否被将军 → 若被将则非法（送将禁止）
7. 撤销临时走棋
8. 若全部通过 → 合法
```

#### 将军检测算法
```
1. 找到对方所有棋子
2. 对每个对方棋子，检查其攻击范围是否包含己方帅/将位置
3. 若任一对方棋子可攻击到帅/将 → 被将军
```

### 3.3 move_gen.rs — 走法生成模块

**职责**：生成所有合法走法，用于搜索引擎

**生成策略**：
```
1. 遍历当前方所有棋子
2. 对每个棋子，根据其类型生成所有可能目标位置
3. 对每个可能目标，调用 rules 模块验证合法性
4. 返回全部合法走法列表
```

**走法排序优化**（提高 Alpha-Beta 剪枝效率）：
```
走法排序优先级（从高到低）：
1. 吃子走法（按被吃棋子价值降序）
2. 中心控制走法
3. 其他走法
```

### 3.4 evaluation.rs — 局势评估模块

**职责**：评估当前棋局优劣，返回评分

**评估函数设计**：

```
总评分 = Σ(红方棋子评分) - Σ(黑方棋子评分)

每个棋子评分 = 子力基础分 + 位置加分表[棋子类型][x][y]
```

**子力基础分表**：

```rust
const PIECE_BASE_VALUE: [i32; 7] = [
    10000, // King（帅/将）
    200,   // Advisor（仕/士）
    200,   // Bishop（相/象）
    400,   // Knight（马）
    900,   // Rook（车）
    450,   // Cannon（炮）
    100,   // Pawn（兵/卒）
];
```

**兵/卒动态评分**：
- 未过河：基础分 100
- 已过河：基础分 200 + 位置加分大幅增加

**位置加分表设计**：

每种棋子类型有 10×9 的位置加分矩阵，值由经验公式和历史对局数据确定。示例（红马位置加分表，简化版）：

```rust
// 红马位置加分（仅在黑方区域加分显著）
const RED_KNIGHT_POS_TABLE: [[i32; 9]; 10] = [
    [  0,  -4,   0,   0,   0,   0,   0,  -4,   0],  // y=0 黑底线
    [  0,   2,   4,   4,  -2,   4,   4,   2,   0],  // y=1
    [  4,   2,   8,  10,  12,  10,   8,   2,   4],  // y=2
    [  2,   6,  14,  16,  14,  16,  14,   6,   2],  // y=3
    [  4,  10,  16,  14,  12,  14,  16,  10,   4],  // y=4 黑前沿
    [  4,  10,  16,  14,  12,  14,  16,  10,   4],  // y=5 红前沿
    [  2,   6,  14,  16,  14,  16,  14,   6,   2],  // y=6
    [  4,   2,   8,  10,  12,  10,   8,   2,   4],  // y=7
    [  0,   2,   4,   4,  -2,   4,   4,   2,   0],  // y=8
    [  0,  -4,   0,   0,   0,   0,   0,  -4,   0],  // y=9 红底线

```

黑方棋子位置加分表为红方的**镜像翻转**（y坐标取反）。

**机动性加分**（可选优化）：
```
机动性加分 = 可走合法位置数 × 6（每方）
```

### 3.5 engine.rs — 搜索引擎模块

**职责**：Minimax + Alpha-Beta 剪枝搜索，返回最优走法

#### Alpha-Beta 搜索算法

```
function alpha_beta(board, depth, alpha, beta, maximizing):
    if depth == 0 or game_over(board):
        return evaluate(board)

    moves = generate_legal_moves(board)
    sort_moves(moves)  // 吃子走法优先

    if maximizing:  // 红方（最大化评分）
        for move in moves:
            board.make_move(move)
            score = alpha_beta(board, depth-1, alpha, beta, false)
            board.undo_move(move)
            alpha = max(alpha, score)
            if alpha >= beta:
                break  // β剪枝
        return alpha
    else:  // 黑方（最小化评分）
        for move in moves:
            board.make_move(move)
            score = alpha_beta(board, depth-1, alpha, beta, true)
            board.undo_move(move)
            beta = min(beta, score)
            if beta <= alpha:
                break  // α剪枝
        return beta
```

#### Transposition Table 优化

```
搜索前查表：
1. 计算当前局面 Zobrist 哈希
2. 查找置换表
3. 若表中有条目且深度 >= 当前搜索深度：
   - Exact: 直接返回评分
   - LowerBound: 用作 alpha 下界
   - UpperBound: 用作 beta 上界
4. 搜索完成后存表
```

#### 走法排序优化

```
排序规则（优先级从高到低）：
1. 置换表中的最佳走法（若命中）
2. 吃子走法（MVV-LVA：最贵受害者-最贱攻击者）
   - 排序分 = 被吃棋子基础分 - 吃子棋子基础分/100
3. 其他走法
```

#### 搜索结果

```
analyze(depth) → AnalysisResult {
    1. 遍历当前方所有合法走法
    2. 对每个走法执行 alpha_beta 搜索
    3. 收集所有走法的评分
    4. 按评分排序，取 Top 3 候选
    5. 记录最优走法的搜索路径
    6. 返回 AnalysisResult
}
```

### 3.6 board.js — 前端棋盘渲染模块

**职责**：棋盘绘制、棋子显示、交互处理

**棋盘渲染方案**：
- 使用 **Canvas 2D** 绘制棋盘网格和棋子
- 棋盘尺寸自适应容器，按比例缩放

**交互处理**：
```
点击事件流程：
1. 点击位置 → 计算棋盘坐标 (x, y)
2. 若当前有选中棋子：
   a. 检查目标是否为合法走法位置
   b. 若合法 → 调用 game.js 执行走棋
   c. 若不合法（目标为己方棋子）→ 切换选中
   d. 若不合法 → 取消选中
3. 若当前无选中棋子：
   a. 检查点击位置是否为己方棋子
   b. 若是 → 选中该棋子，显示合法目标位置
   c. 若否 → 忽略
```

**渲染层次**：
```
绘制顺序（由底到顶）：
1. 棋盘背景（木质纹理）
2. 网格线 + 河界 + 九宫斜线
3. 上一步走法标记（黄色高亮）
4. 推荐走法箭头（绿色箭头）
5. 合法目标位置提示（半透明圆点）
6. 棋子
7. 选中棋子高亮边框
```

### 3.7 game.js — 游戏状态管理模块

**职责**：管理游戏状态、协调UI与引擎交互

**状态管理**：
```
GameState {
    engine: XiangQiEngine (WASM实例，通过Worker访问)
    selected_piece: Option<Position>
    last_move: Option<Move>
    is_analyzing: boolean
    analysis_depth: u8 (默认4)
    move_history: Vec<MoveRecord>
    game_over: Option<Side>
}
```

**核心操作**：
- `newGame()`：重置引擎和UI
- `makeMove(from, to)`：验证走棋合法性 → 执行走棋 → 更新UI → 检查胜负
- `undoMove()`：撤销走棋 → 更新UI
- `analyze()`：发送分析请求到 Worker → 接收结果 → 更新分析面板
- `importFEN(fen)`：解析FEN → 重建引擎 → 更新UI
- `exportFEN()`：从引擎导出FEN

### 3.8 worker.js — Web Worker 模块

**职责**：在后台线程加载 WASM 并执行引擎计算

**初始化流程**：
```javascript
// worker.js
let engine = null;

self.onmessage = async function(e) {
    if (!engine) {
        // 首次收到消息时初始化 WASM
        const wasmModule = await import('./pkg/xiangqi_core.js');
        await wasmModule.default();  // 初始化 WASM
        engine = new wasmModule.XiangQiEngine();
    }

    const { type, data } = e.data;

    switch (type) {
        case 'analyze':
            engine = reconstructFromData(engine, data.boardState);
            const result = engine.analyze(data.depth);
            self.postMessage({ type: 'analysis_result', data: result });
            break;
        case 'validate_move':
            const valid = engine.make_move(data.fromX, data.fromY, data.toX, data.toY);
            self.postMessage({ type: 'validation_result', data: { valid } });
            break;
        // ... 其他操作
    }
};
```

**注意**：WASM 实例不能跨线程传递，Worker 中需独立维护引擎实例。每次操作前需同步棋盘状态。

### 3.9 main.js — 入口初始化模块

**职责**：DOM 初始化、模块协调、事件绑定

**初始化流程**：
```
1. 创建 Game 实例
2. 创建 Canvas 棋盘渲染器
3. 创建 Web Worker
4. 绑定按钮事件（新局、悔棋、分析等）
5. 绑定 Canvas 点击事件
6. 绑定 Worker 消息回调
7. 渲染初始棋盘
```

## 4. FEN 格式设计

### 4.1 FEN 规则

象棋 FEN 格式参考国际象棋 FEN，适配中国象棋规则：

```
rnbakabnr/9/1c5c1/p1p1p1p1p/9/9/P1P1P1P1P/1C5C1/9/RNBAKABNR w - 0 1
```

| 字段 | 说明 |
|------|------|
| 棋盘部分 | 从上到下（黑方底线→红方底线），每行用 `/` 分隔，数字表示连续空格数 |
| 棋子字母 | 红方大写：R车、N马、B相、A仕、K帅、C炮、P兵；黑方小写 |
| 走棋方 | `w`=红方，`b`=黑方 |
| 半回合数 | 双方未吃子走棋总数（暂不使用，填0） |
| 回合数 | 当前回合编号 |

### 4.2 解析与生成

- `from_fen(fen)`：解析 FEN 字符串 → 重建棋盘
- `to_fen()`：遍历棋盘 → 生成 FEN 字符串

## 5. 性能优化设计

### 5.1 Rust/WASM 层优化

| 优化项 | 方法 | 预期效果 |
|--------|------|---------|
| Alpha-Beta 剪枝 | 搜索时剪除无用分支 | 减少 50-70% 搜索节点 |
| 置换表 | 缓存已评估局面 | 避免重复评估，深度搜索加速 |
| 走法排序 | 吃子走法优先搜索 | 提高剪枝效率 |
| Zobrist 哈希 | 增量更新而非全量计算 | 走棋时O(1)更新哈希 |
| 帅将位置缓存 | 直接读取而非遍历 | 飞将检测O(1) |
| 静态评估 | 基于位置加分表而非动态计算 | 评估O(棋子数)，极快 |

### 5.2 前端层优化

| 优化项 | 方法 | 预期效果 |
|--------|------|---------|
| Web Worker | WASM 计算在后台线程 | UI 不冻结 |
| Canvas 渲染 | 仅重绘变化区域 | 60fps流畅 |
| 异步消息 | Worker 通信异步化 | 分析时UI仍可响应 |
| 状态缓存 | 前端缓存棋盘状态 | 减少WASM调用次数 |

### 5.3 WASM 编译优化

Cargo.toml 编译优化配置：
```toml
[profile.release]
opt-level = 3          // 最高优化等级
lto = true             // 链接时优化
codegen-units = 1      // 单编译单元，更好的优化
strip = true           // 去除调试信息，减小体积
```

## 6. 错误处理设计

### 6.1 Rust 层错误处理

- 走棋合法性：返回 `bool`（合法/非法）
- FEN 解析：返回 `Result<T, JsValue>`，非法 FEN 返回错误信息
- 游戏结束：返回 `Option<u8>`（胜负方）
- 搜索超时：设置搜索时间上限，超时返回当前最优结果

### 6.2 前端层错误处理

- Worker 通信失败：显示提示信息，建议刷新
- WASM 初始化失败：降级提示，建议使用现代浏览器
- 无效操作（如未选中棋子时走棋）：静默忽略

## 7. 安全设计

- 系统为纯前端应用，无网络通信，无安全风险
- WASM 在浏览器沙箱中执行，无系统访问权限
- 无用户数据收集，无隐私问题
